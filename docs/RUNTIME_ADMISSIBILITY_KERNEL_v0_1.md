# Runtime Admissibility Kernel v0.1

## Purpose

This document describes runtime admissibility inside KuuOS.

## Core Clarification

Capability does not automatically imply admissibility.

KuuOS distinguishes:

- what can be generated
- what can be routed
- what can be admitted into runtime operation

## Runtime Admissibility Conditions

Admissibility may require:

- provenance visibility
- governance visibility
- claim-level clarity
- rollback capability
- uncertainty visibility
- boundary preservation
- fail-closed capability

## Runtime Non-Admissibility

A runtime surface may be non-admissible even if:

- fluent
- plausible
- internally coherent
- highly capable

## Preferred Runtime Outcomes

When admissibility conditions are incomplete, KuuOS prefers:

- hold
- abstain
- rollback
- reobserve
- handover

rather than unsafe escalation.

## Relation to Runtime Governance

Runtime governance evaluates admissibility continuously rather than assuming permanent validity.

## Relation to Provenance

Loss of provenance may reduce admissibility.

## Interpretation

This kernel defines runtime governance orientation.

It does not itself grant theorem, institutional, deployment, or clinical authority.
