    CodeAI Generated Patch Error Reduction Comparative Replay Evaluation v0.1¶

    Purpose¶

    This additive post-roadmap stage closes the measurement loop opened by CodeAI Generated Patch Error Baseline Replay Evaluation v0.1.

    It consumes two independently completed, read-only replay evaluations:

    1. the fixed baseline evaluation;
    2. a content-addressed successor evaluation produced after the nine-stage generated-patch error-reduction roadmap.

    It compares their sealed metrics deterministically and emits either:

    • generated_patch_error_reduction_confirmed; or
    • generated_patch_error_reduction_not_confirmed.

    A valid comparison that misses one or more thresholds is still a completed measurement. It is not converted into rejection, repair authority, or an automatic next action.

    Why this stage follows roadmap step 9¶

    The baseline specification requires every planned successor to be evaluated against the same replay dataset or a content-addressed successor dataset. Steps 1–9 are now implemented:

    1. Baseline and Replay Evaluation
2. Selective Repository Semantic Context Pack
3. Typed Structured Edit IR
4. Candidate Static Admissibility Preflight
5. Typed Error Classification
6. Independent Test Strengthening
7. Evidence-Weighted Selection and Abstention
8. Version-Bound Repair Memory
9. Maintainability Trajectory Gate

    The comparative replay stage does not add a tenth generation mechanism. It determines whether the completed roadmap improved the measured pipeline under an explicit deny-by-default comparison policy.

    Exact inputs¶

    The kernel requires:

    1. one sealed baseline replay evidence object;
    2. its exact sealed route receipt;
    3. one sealed successor replay evidence object;
    4. its exact sealed route receipt;
    5. a fresh comparison request;
    6. a sealed comparison policy.

    Both source evaluations must use the canonical CodeAI Generated Patch Error Baseline Replay Evaluation v0.1 profile and must preserve:

    • exact source correspondence;
    • completed read-only evaluation;
    • no historical code re-execution;
    • no provider or runner invocation;
    • no repository or Git effect;
    • no network or secret access;
    • no selection or successor authority;
    • no correctness claim.

    The request and policy bind all four source digests and the same repository.

    Dataset relation¶

    The reference policy requires:

    baseline dataset digest != successor dataset digest
baseline case count = successor case count

    A successor dataset is therefore content-addressed independently while preserving comparison size. This profile does not claim that either dataset is complete, representative, random, or unbiased.

    Comparative metrics¶

    The stage computes signed successor - baseline deltas for:

    • conditional pass rate at each sequential stage;
    • independent-verification conditional pass rate;
    • first-failure counts;
    • independently verified patch count;
    • incomplete case count;
    • repair-green rate;
    • repair cycles per verified patch;
    • provider calls per verified patch;
    • generated output bytes per verified patch;
    • distinct and repeated normalized error fingerprints;
    • cases containing a repeated error fingerprint.

    Rate and efficiency deltas use integer basis points from the source replay metrics. Undefined source ratios fail closed when the policy requires defined comparisons.

    Reference comparison¶

    The committed fixture compares three baseline cases with three successor cases.

    Measurement Baseline Successor Delta
    Independently verified patches 2 3 +1
    Typecheck first failures 1 0 -1
    Repeated error fingerprints 1 0 -1
    Cases with repeated error fingerprints 2 0 -2
    Repair cycles per verified patch 5000 bp 3333 bp -1667 bp
    Provider calls per verified patch 30000 bp 16666 bp -13334 bp
    Generated bytes per verified patch 15000000 bp 8833333 bp -6166667 bp

    The independent-verification conditional pass rate remains 10000 bp among cases that reached verification, while verified coverage expands from two to three patches. The policy therefore requires no pass-rate regression and a verified-patch count increase of at least one.

    All eight reference targets are met, so the fixture disposition is generated_patch_error_reduction_confirmed.

    Confirmation policy¶

    Confirmation requires all of:

    1. independent-verification conditional pass-rate delta at or above its floor;
    2. verified-patch count delta at or above its floor;
    3. typecheck first-failure delta at or below its ceiling;
    4. repeated-fingerprint delta at or below its ceiling;
    5. repeated-error-case delta at or below its ceiling;
    6. repair-cycle efficiency delta at or below its ceiling;
    7. provider-call efficiency delta at or below its ceiling;
    8. generated-output efficiency delta at or below its ceiling.

    Thresholds are explicit signed integers. No historical memory hint, trajectory admission, or prior success can waive them.

    Outcomes¶

    Confirmed¶

    All exact-lineage and policy checks pass and every target is met.

    error_reduction_confirmed = true

    This is bounded evidence of improvement on the supplied replay comparison. It is not a correctness proof, probability statement, or future guarantee.

    Not confirmed¶

    All inputs are valid and comparison completes, but at least one target is not met.

    error_reduction_confirmed = false

    This is a measured non-confirmation, not rejection and not repair authority.

    Blocked¶

    Malformed shape, missing or extra fields, invalid source digest, source-effect claim, stale request, unauthorized evaluator, repository mismatch, source substitution, identical dataset digest, unequal required case count, undefined required ratio, or forbidden policy capability returns blocked with no evidence or receipt.

    Fixed boundaries¶

    comparative replay != historical code execution
comparison result != correctness proof
comparison result != probability
comparison result != unbiased-dataset proof
metric improvement != universal software quality
not confirmed != rejection
not confirmed != repair requirement
confirmation != candidate-selection authority
confirmation != execution authority
confirmation != Git authority
comparison receipt != successor-stage authority

    The kernel performs no provider call, runner invocation, repository mutation, Git effect, network access, secret read, selection, repair, execution, merge, or deployment.

    Formal surface¶

    The generic Lean kernel defines:

    • a bounded replay metric vector;
    • signed successor-minus-baseline deltas;
    • a threshold policy;
    • the ReductionConfirmed proposition;
    • extraction theorems for every confirmed threshold;
    • explicit downstream authority and epistemic boundaries.

    The actual specialization proves by computation that the reference baseline and successor vectors satisfy all eight targets while every effect and overclaim flag remains false.

    Validation¶

    The dedicated workflow validates:

    • Python syntax;
    • example and manifest JSON;
    • deterministic fixture reconstruction;
    • 23 dedicated confirmation, non-confirmation, exact-lineage, tamper, and fail-closed tests;
    • baseline replay and maintainability-gate regressions;
    • canonical Lake manifest;
    • strict dedicated Lean root;
    • strict full KuuOSFormal.

    Failure diagnostics are uploaded only after a completed failing job.
